# The three plays

## Play 1 — Understand the work

### Solves for
- I inherited this team or repo.
- I missed context from earlier conversations.
- I need to prepare for a leadership check-in.
- I do not know what this project actually does.

### Where to use it
GitHub.com, inside the shared Space.

### Prompt
> Explain this repository and current workstream in plain English for a manager.  
> Tell me what the system does, what looks actively changing, where delivery risk may exist, and what I should ask the team next.

### Output you want
- plain-English summary
- active workstream
- likely risks
- smart next questions

### Save back
If the answer is useful, capture the key points in a short Markdown note for the Space source set.

---

## Play 2 — Shape the work

### Solves for
- I have messy notes from a meeting.
- I have a screenshot or a rough ask.
- I need a clean issue that engineering can act on.

### Where to use it
GitHub.com issue creation or Copilot Chat.

### Prompt
> Turn these notes into a clear GitHub issue.  
> Include the problem statement, desired outcome, acceptance criteria, assumptions, and risks or dependencies.  
> If something is missing, list open questions instead of guessing.

### Output you want
- clear problem statement
- desired outcome
- acceptance criteria
- assumptions
- risks / dependencies
- open questions

### Save back
Submit the issue only after a human review.

---

## Play 3 — Review the work

### Solves for
- I need to understand a PR quickly.
- I want better review questions.
- I want visibility without pretending to be the code author.

### Where to use it
GitHub.com PR page.

### Prompt
> Summarize this pull request for a manager.  
> Focus on user impact, release or operational risk, dependencies, and two or three smart review questions I should ask.

### Output you want
- what changed
- who or what is affected
- risks
- dependencies
- questions to ask

### Save back
If the review question list is strong, add the best ones to your team's manager review template.
